## 1. Overview

`ghostlogic-agent-watchdog` ships the `logicd` daemon — a forensic forwarder for AI-coding sessions (Claude Code, Codex CLI). It watches session JSONL transcripts, tick-batches events, hash-chains an audit log, and POSTs signed batches to GhostLogic Blackbox. Conservative privacy defaults (`include_payload=false`), explicit `EXIT_AUTH_FATAL=3` on 401/403, durable offset commits per file. Apache 2.0. Uses `spine.Core` (workspace SDK pattern). Code quality is high for an 0.1.0 release.

- **Repo:** `github.com/adam-scott-thomas/ghostlogic-agent-watchdog` (live remote)
- **Tests:** 47 passing, 2 skipped (intentional platform skips, per failed agent's run)
- **Source layout:** `logicd/{__main__.py, audit.py, batch.py, client.py, config.py, enroll.py, installer.py, redact.py, state.py, tick.py, watcher.py, adapters/}`

### F-WD-002 (HIGH) `fp:wd-endpoint-id-stored-not-shipped`
**`endpoint_id` is captured at enrollment but never surfaces on outbound payloads.**

`logicd/enroll.py:75, 149, 155` — enrollment writes `endpoint_id = "{endpoint_id}"` into the config TOML and persists it. But in steady-state shipping:

- `logicd/adapters/base.py:16, :52` — uses `socket.gethostname()` for source identity, no endpoint_id reference.
- `logicd/client.py:53, :61, :63` — `ship` is called with `hostname=`; payload sets `source_id: hostname` and `endpoint_name: hostname`. No `endpoint_id` field.
- `logicd/watcher.py:85, :182` — adds `hostname` to telemetry, no `endpoint_id`.

Same gap pattern as the logicd 2026-05-01 audit F-004. The endpoint-isolation v1 layout exists at `C:\ProgramData\GhostLogic\` and this code reads it during enroll, but **the canonical UUID never reaches Blackbox in shipping events**. Multi-tenant per-endpoint correlation on the server side relies on the more-collidable `socket.gethostname()`.

**Remediation:** add `endpoint_id` to the `IngestPayload` shape and the `ship` call signature. Plumb it through `Forwarder` from `Config`.

### F-WD-003 (MEDIUM) `fp:wd-apikey-toml-template`
**`logicd/enroll.py:73-75` writes the API key into the config TOML in plaintext.**

```python
key = "{api_key}"
endpoint_id = "{endpoint_id}"
```

These are template-string substitutions — the format-string is then expanded with the actual key and written to disk at the operator's `logicd.toml` (production: `C:\ProgramData\GhostLogic\agents\logicd.toml`).

Same finding pattern as Agent_Rust F-AGR-004 and logicd 2026-05-01 F-001. Workspace policy is 1Password.

**evidence_note:** I did not read the deployed `logicd.toml` contents in this round; the logicd audit (same date, separate report) did read it and confirmed a live `gl_agent_*` key in plaintext. That finding is **the same physical file** as this watchdog repo would write to on enrollment.

**Remediation:** post-enrollment, write the API key to the OS keyring (Windows Credential Manager via `keyring` lib, or DPAPI directly). The TOML stores only a reference (`api_key_keyring_id = "logicd_default"`).

### F-WD-005 (MEDIUM) `fp:wd-include-payload-default-only`
**Privacy default `include_payload=False` is correct, but only conservative-by-config — no compile-time or schema-level enforcement.**

`logicd/config.py:36-52` sets `include_payload=False` as the default. Operators who set `include_payload=true` ship full transcript content (which can contain secrets, regardless of `redact_patterns`). The redact pattern list is best-effort regex.

**Severity rationale:** MEDIUM because the default is safe — but for a forensic forwarder shipping AI-coding transcripts, "best-effort regex over user content" is a thin defense if `include_payload` is ever flipped on without a thorough secret-pattern review.

**Remediation:** require operators to acknowledge `include_payload=true` via an explicit `i_understand_this_ships_full_transcripts = true` second flag, or via a CLI confirmation on first enable. Document the residual risk in README.
